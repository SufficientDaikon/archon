OMNISKILL Architecture

Understand the layered architecture, design principles, and data flow that power OMNISKILL's universal approach.

🎯 Design Principles

OMNISKILL is built on five core principles that guide every design decision:

  1. Universal Format
    Write once, deploy everywhere. Skills, agents, and pipelines are defined in a universal format that adapters transform for each platform.
  2. Composability
    Skills compose into bundles. Agents compose into pipelines. Everything is designed to be combined and reused in new contexts.
  3. Separation of Concerns
    Skills contain knowledge. Agents provide orchestration. Pipelines define workflows. Each layer has a clear, distinct responsibility.
  4. Self-Improvement
    The system uses its own tools to improve itself. The skill factory creates skills. Pipelines improve pipelines. Agents maintain agents.
  5. Platform Agnostic
    Platform differences are abstracted by adapters. Authors focus on what skills do, not how each platform implements them.

🔄 Data Flow

How a user request flows through the system:

1. User Request → "Build feature X from scratch"
↓
Trigger matching → Matches "sdd-pipeline"
2. Pipeline Activation → SDD Pipeline starts
↓
Step 1: spec-writer agent activated
3. Agent Selection → spec-writer loads
↓
Skill binding → Uses "spec-writing" skill
4. Skill Execution → Skill provides instructions
↓
Resources loaded → spec-template.md, best-practices.md
AI model executes → Follows instructions + resources
5. Output Generation → spec.md created
↓
Validation → Output contract checked
Artifact saved → spec.md written to disk
6. Handoff → Next agent (implementer) receives spec.md
💡
Key Insight

Each layer adds value without knowing about layers above it. Skills don't know they're in a bundle. Bundles don't know they're in a pipeline. This separation enables maximum reuse.

🔌 Adapter System

Adapters bridge the gap between universal skills and platform-specific formats:

Adapter Responsibilities

  • Format transformation: Convert SKILL.md to platform-native format
  • Metadata handling: Extract and format trigger keywords, tags, etc.
  • Resource packaging: Bundle resources in platform-appropriate way
  • Override merging: Combine base skill with platform overrides
  • Validation: Ensure output meets platform requirements

Adapter Pipeline

Universal Skill (SKILL.md + manifest.yaml)
         ↓
    Load base content
         ↓
    Load platform override (if exists)
         ↓
    Merge base + override
         ↓
    Transform to platform format
         ↓
    Package resources
         ↓
    Validate output
         ↓
    Write to platform directory
         ↓
Platform-Specific Skill (e.g., .mdc for Cursor)
✓
Adding New Platforms

To support a new platform, create an adapter in adapters/ that implements the standard adapter interface. No changes to skills, bundles, agents, or pipelines needed.

♻️ Self-Improvement

OMNISKILL uses its own capabilities to maintain and improve itself:

What Tool Used How
Create new skills Skill Factory Pipeline Interactive workflow guides skill creation
Improve existing skills skill-improvement agent Analyzes usage, suggests improvements
Test skills Validation framework Automated test cases for all skills
Document features documentation-writer skill Generates docs from code/manifests
Debug pipelines Debug Pipeline Investigates, diagnoses, fixes issues
💡
Meta-Circular Evolution

This self-improvement capability creates a positive feedback loop. Better tools make it easier to create better tools, which make it easier to create even better tools.

🔧 Extensibility Points

OMNISKILL is designed to be extended at multiple levels:

1. Skills

Add new skills by copying the template and following the guide.

2. Bundles

Group related skills with a meta-skill for intelligent routing.

3. Agents

Define new orchestration personas with skill bindings and handoff protocols.

4. Pipelines

Compose agents into multi-step workflows with branching and loops.

5. Adapters

Support new platforms by implementing the adapter interface.

6. Validators

Add custom validation rules for domain-specific requirements.
